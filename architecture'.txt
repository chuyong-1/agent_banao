High-Velocity Live Planning Runtime (LangGraph Pipeline)When a natural-language staffing request enters the system via the MCP stdio interface (server.py), the request payload triggers a linear execution path through four sequential LangGraph processing steps:Node 1: Intent Extraction (extract_intent_node)Receives the unstructured natural language string inside the GraphState repository.Employs a split parsing strategy: attempts to use an LLM-driven structured extraction framework with a local validation cache to accelerate processing. If a timeout occurs or the connection drops, it smoothly fails over to a local regex fallback parser (_heuristic_extract).Extracts and structures project requirements like duration windows, resource counts, required skill clusters, and department boundaries.Node 2: Enterprise Resource Data Ingestion (ingest_erp_data_node)Pulls the structured output parameters compiled by Node 1.Avoids out-of-process file calls by establishing a direct transactional connection to the Tier 1 operational database tables via db.py.Loads rows from people, assignments, leaves, and bounties that correspond to the active tracking window, packing them into an efficient dictionary format within the running graph context.Node 3: Algorithmic Availability & Skill Calculation (compute_metrics_node)This node processes the raw database fields using several key calculations:Word-Boundary Regex Skill Matching: To prevent false positives (e.g., matching the token "Go" with "Django" or "Java" with "JavaScript"), the matching utility uses explicit lookbehind and lookahead word-boundary assertions ((?<!\w) and (?!\w)). This guarantees precise alignment for delicate names like C++, .NET, and F#.Urgency-Weighted Capacity Drain: Instead of assuming a generic time allocation for outstanding tasks, the engine calculates real-time resource availability by checking the urgency of open bounties using this calculation:$$\text{Bounty Capacity Drain} = \frac{\text{Hours Estimated}}{\max\left(0.5, \frac{\text{Days Remaining}}{7}\right)}$$This ensures that an unresolved task due tomorrow applies a much larger immediate capacity drain than a task due in several weeks.Interval-Merging Overlap Calculation: To figure out exactly how much time an employee will miss due to vacation, the engine runs a merging check on leave records before summing the totals. This prevents overlapping vacation entries from falsely double-counting the amount of unavailable time.Node 4: Resource Matchmaker Ranking (matchmaker_node)Aggregates the capacity numbers and calculated skill scores to compute a final fit_score and reliability_score for each candidate.Applies strict defensive screening logic: any employee whose available capacity drops exactly to zero is immediately routed into a disqualified_candidates list with a status label of HARD_DISQUALIFY.Groups the remaining eligible candidates by team suitability, matches them against the project constraints, and outputs a ranked allocation dashboard back to the server response payload.3. The Trajectory Penalty AlgorithmThe engine evaluates employee performance using a rigid, penalty-based evaluation structure. Employees start with a base score of 100, which drops according to specific performance drops tracked by the upstream CELL agent:$$\text{Growth Score} = 100 - \text{Penalties}_{\text{Task Completion}} - \text{Penalties}_{\text{Task Slippage}} - \text{Penalties}_{\text{Compliance}}$$Task Completion Deficit: Deducts up to 30 points if task completion rates fall below target baselines.Task Slippage (Carry Rate): Deducts up to 20 points based on how often deadlines are missed or pushed to the next sprint.Process Compliance: Deducts up to 20 points if an employee falls behind on required administrative steps, such as missing daily End-of-Day (EOD) logs.Time-Freezing Leave AdjustmentWhen calculating career tracking dates (e.g., verifying if an intern has reached their 6-month evaluation milestone), the pipeline executes a protective adjustments pass. It checks the employee's history for any approved leaves of absence, subtracts those non-working days, and extends the evaluation window accordingly. This "time-freezing" mechanic ensures that extended medical or personal leave doesn't penalize an employee's career progression clock.Automated Leech Detection LoopThe script reviews historical snapshots over a rolling 6-month execution window. If an employee's career path remains stuck in an introductory phase (like intern_hybrid) beyond the threshold duration while displaying a flat or declining trajectory score trend, the engine registers a LeechFlag in the database.Real-Time Alerts LayerWhen the evaluation engine generates a StageGateFlag or a LeechFlag, it automatically calls a structured notification handler (push_slack_alert). This formats the system flags into a readable notification payload and posts it directly to an external Slack webhook channel, alerting human resource managers immediately when an employee requires direct attention.Immutable System LoggingEvery analytical run concludes by appending a status log entry to the stroma_actions ledger table. This table stores the exact state configurations inside an append-only JSONB field, creating a reliable audit trail that the internal HRMS chatbot can scan to explain why past decisions were made.4. Database Schema Design (Tiers Definition)The database schema (models.py) uses a relational tier structure optimized for PostgreSQL 14+. This setup decouples high-frequency live table reads from long-term time-series logging:Tier 1: Core Operational TablesEntities: departments, people, assignments, leaves, bounties, skill_profiles.Structural Mechanics: These tables provide the raw, real-time data read by the LangGraph pipeline during staffing requests. They are decoupled from external enterprise resource systems by mapping relationships using clean, unique string identifiers (employee_id) rather than internal database UUID fields.Tier 2: Analytical SnapshotsEntities: growth_snapshots, capacity_snapshots.Structural Mechanics: Written exclusively by background cron operations. These tables leverage flexible PostgreSQL JSONB document columns to store complex structural summaries, such as bench_metrics and workforce_health, without requiring ongoing database schema migrations as the data models evolve.Tier 3: Lifecycle Tracking & Audit LedgerEntities: stage_transitions, stage_gate_flags, assessment_results, hiring_flags, leech_flags, stroma_actions.Structural Mechanics: This layer houses the strict compliance logs and systemic flags generated by the STROMA engine. Database-level constraints (CheckConstraint) rigorously protect the integrity of the state machine values, while tables like stroma_actions serve as an append-only, tamper-proof repository tracking all automated system choices.5. Architectural Health Verification SafeguardsTo prevent configuration errors or software bugs from breaking production workflows, your testing architecture includes several validation suites:Strict Substring Deficit Protection (test_pipeline.py): Runs regression tests against the word-boundary filtering regex. This confirms that searching for common short strings like "Go" won't mistakenly match broader terms like "Django", keeping technical allocations precise.State Serializability Verification (test_bench_health.py): Runs continuous data transformation loops that convert nested database elements into structured Pydantic objects and back into JSON strings. This guarantees that complex database fields load and save cleanly without corruption.High-Load Scaling Verifications (stress_test.py): Simulates massive organization sizes by running synthetic data updates simultaneously across 1,000 employee records. This verifies that the processing code handles large datasets efficiently while staying within safe memory limits.Race-Condition Threading Checks: Executes multiple simulated allocation adjustments concurrently to stress-test the transactional isolation levels. This confirms that the database session factory correctly prevents record overwrites or locking conflicts under heavy multi-user traffic.
